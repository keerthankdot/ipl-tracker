## Operating Principles

These are Ascnd's core building principles. They govern every decision in this project, from architecture to copy to what we ship and what we kill. Read these before writing a single line of code.

### 1. Empire State Building

You don't build the Empire State Building by starting with the penthouse. You start with the foundation, then steel, then floors, then the spire. Every phase of Win Path follows this: Phase 1 is the foundation (data + basic sim). Phase 2 is the steel (weighted sim + venue). Phase 3 is the floors (AI narrative + NRR). Phase 4 is the spire (viral + monetization). Never skip ahead. Never build a flashy feature on a shaky data layer. If the NRR formula is wrong, the AI narrative on top of it is worthless. Get each layer right before stacking the next one. When in doubt, ask: "Is the floor below this solid?"

### 2. Creative Challenger, Not a Blind Validator

Do NOT just execute instructions robotically. Challenge assumptions. If something in this CLAUDE.md looks wrong, suboptimal, or could be done better, SAY SO. If the venue data schema is missing a critical field, flag it. If the Monte Carlo approach has a known flaw, propose an alternative. If a component is over-engineered for the MVP, cut it. The builder's job is to think critically, push back on bad ideas (even if they came from this document), and propose better solutions. "Yes and here's why we should do it differently" is always better than silent compliance. This product wins by being smarter than the competition, not by following a spec blindly.

### 3. Robert Downey Jr. Rule

Robert Downey Jr. was written off. Career over. Done. Then he became Iron Man and the highest-paid actor in Hollywood. The rule: never write off a possibility, never assume the current state is the final state, and always build for the comeback scenario. In product terms: never assume the first approach will work. If the Sportmonks API is garbage, pivot to scraping. If Monte Carlo is too slow for real-time, switch to pre-computed lookup tables. If the AI narrative sounds robotic, rewrite the prompt 10 times until it sounds human. Every problem has a second act. The first attempt failing is not failure. Giving up after the first attempt is.

### 4. Be Objective, Ask Questions, Learn

Before building anything, understand WHY it exists. Don't just implement a "Venue Intelligence" panel because the CLAUDE.md says so. Ask: does the user actually need altitude data? Will a casual RCB fan care about spin wicket percentage? What's the minimum viable version of this panel that delivers value? Research before building. Validate assumptions with real data. If IPL historical venue stats show that home advantage is only 52% (not 55%), use 52%. If player form data from the API is unreliable, don't build a feature on unreliable data and pretend it's accurate. Objectivity over optimism. Questions over assumptions. Learning over guessing. When you don't know something, search for the answer before making it up.

### 5. Always Price at the Higher End

When there's a choice between building something basic or building something premium, build premium. Win Path is not a free hobby project. It's an Ascnd product. The share cards should look like they were designed by a studio, not generated by a template. The AI briefings should read like a top cricket analyst wrote them, not like ChatGPT summarized a stats page. The dashboard should feel like a Bloomberg Terminal for cricket, not a Bootstrap admin panel. Premium positioning from day one. If someone screenshots this product and posts it, it should look like it costs money even on the free tier. This applies to code quality too: clean architecture, typed interfaces, proper error handling. Build it like you're charging for it, because you are.

## IPL 2026 Format Rules

These are non-negotiable cricket facts. Hard-code them.

- 10 teams, 74 league matches, 4 playoff matches (Qualifier 1, Eliminator, Qualifier 2, Final)
- Each team plays 14 league matches (split across 2 virtual groups: play teams in own group once, teams in other group twice)
- Points: Win = 2 pts, Loss = 0 pts, No Result (rain) = 1 pt each, Tie = decided by Super Over
- Top 4 on points qualify for playoffs. NRR breaks ties when points are equal.
- Qualifier 1: Rank 1 vs Rank 2 (winner to Final, loser to Qualifier 2)
- Eliminator: Rank 3 vs Rank 4 (loser eliminated)
- Qualifier 2: Loser of Q1 vs Winner of Eliminator (loser eliminated)
- Final: Winner of Q1 vs Winner of Q2
- 16 points (8 wins) is historically the safe qualification threshold. 14 points is borderline. 12 points has qualified only once (SRH 2019).
- Season: March 28 to May 31, 2026. 13 venues across India.

### NRR Formula

```
NRR = (Total runs scored / Total overs faced) - (Total runs conceded / Total overs bowled)
```

- If a team is bowled out, overs faced = full 20.0 (not the actual overs used)
- If a team wins chasing, overs faced = actual overs used (e.g., 18.2 overs)
- NRR is CUMULATIVE across all matches in the season, not per-match
- A team that wins by big margins AND defends/chases quickly builds a strong NRR buffer

## Simulation Engine (Python Backend)

### Monte Carlo Core (simulator.py)

```python
# Pseudocode for the simulation engine

def simulate_season(current_standings, remaining_fixtures, weights, n_simulations=50000):
    """
    Run n_simulations of the remaining season.
    For each simulation:
      1. For each remaining match, determine winner based on weighted probability
      2. Generate a realistic scoreline (not just W/L) for NRR calculation
      3. Update points table and NRR after each match
      4. At end of season, record final standings (top 4, top 2, etc.)
    
    Return: For each team, % of simulations where they finish Top 4, Top 2, Win Title
    """
    results = {team: {"top4": 0, "top2": 0, "title": 0} for team in teams}
    
    for sim in range(n_simulations):
        table = copy(current_standings)
        
        for match in remaining_fixtures:
            win_prob = calculate_weighted_probability(match, weights)
            winner = random_choice(match.teams, p=[win_prob, 1 - win_prob])
            scoreline = generate_scoreline(match, winner)  # e.g., 185/6 in 20 vs 172/8 in 19.2
            update_table(table, match, winner, scoreline)
        
        # Rank by points, then NRR
        ranked = rank_teams(table)
        for i, team in enumerate(ranked[:4]):
            results[team]["top4"] += 1
        for i, team in enumerate(ranked[:2]):
            results[team]["top2"] += 1
        
        # Simulate playoffs for title probability
        title_winner = simulate_playoffs(ranked[:4])
        results[title_winner]["title"] += 1
    
    # Convert counts to percentages
    return {team: {k: v / n_simulations * 100 for k, v in stats.items()} for team, stats in results.items()}
```

### Weight Calculation (weights.py)

Each match gets a win probability for each team based on these factors:

```
Base:       50% each (default)
+/- Team Strength:    Dynamic Elo rating (updated after each match, weighted by margin)
+/- Venue:            Home advantage (~55% historically). Venue-specific modifiers.
+/- Form:             Last 3-5 match results. Win streak = +boost, loss streak = -penalty.
+/- Head-to-Head:     Historical record between these 2 teams (light weight, ~5%)
+/- Weather:          Dew probability shifts chase advantage. Rain = NR scenario at X%.
```

Final probability is clamped between 20% and 80%. No match is ever "guaranteed."

### NRR Scoreline Generation (nrr.py)

This is the hard and novel part. For each simulated match, generate a plausible scoreline:

```python
def generate_scoreline(match, winner):
    """
    Generate realistic match scoreline based on:
    - Venue average scores (from venues.json)
    - Team batting/bowling strength indices
    - Whether team batted first or second
    
    Returns: (team1_score, team1_overs, team2_score, team2_overs)
    
    Logic:
    - Base: venue_avg_1st_innings +/- random variance (std dev ~20 runs)
    - Winner's score = base + margin_adjustment
    - Loser's score = base - margin_adjustment
    - Margin drawn from historical distribution (mean ~15 runs, std ~20)
    - If chasing, winner's overs = 20 - (surplus runs / ~8 run rate) with noise
    - If bowled out, loser's overs = 20.0 (full allocation)
    """
```

### Layer 4: Weather + Dew (OpenWeatherMap)

```python
# Dew prediction model
def predict_dew(temperature_c, humidity_pct, match_time_ist):
    """
    Heavy dew: humidity > 70% AND temperature < 25C after sunset (~6:30 PM)
    Moderate dew: humidity > 60% AND temperature < 28C after sunset
    Light/none: otherwise
    
    Evening matches (7:30 PM start) almost always have dew in April-May.
    Afternoon matches (3:30 PM start) rarely have dew.
    """
    is_evening = match_time_ist >= "19:00"
    if is_evening and humidity_pct > 70 and temperature_c < 25:
        return "heavy"
    elif is_evening and humidity_pct > 60 and temperature_c < 28:
        return "moderate"
    return "light"
```

## Critical Rules

### The Principles in Practice
These rules are the Operating Principles (Section 2) applied to specific decisions. If a rule here conflicts with a principle, the principle wins. If you're unsure about an edge case, go back to Empire State Building (is the foundation solid?), Creative Challenger (is there a better way?), RDJ Rule (is there a second act if this fails?), and Objectivity (do I actually know this or am I guessing?).

### Never Do
- Never use gambling language: "bet", "wager", "odds" (in betting context), "punt", "stake"
- Never present predictions as certainties. Always ranges and probabilities.
- Never hard-code match results. Everything flows from API data.
- Never show stale data without a "last updated" timestamp
- Never break NRR calculation. Get the formula right or don't show it. (Empire State Building: this is foundation-level. If NRR is wrong, everything above it crumbles.)
- Never use em dashes in any copy (Keerthan's style rule)
- Never use generic AI design (purple gradients, Inter font, cookie-cutter cards). (Price at the Higher End: every pixel should look intentional and premium.)
- Never blindly implement a feature without asking "does this actually help the fan?" (Be Objective: question everything.)
- Never ship a broken layer and build on top of it. Fix it first. (Empire State Building.)
- Never give up on an approach after one failure. Try 3 different ways before declaring something impossible. (RDJ Rule.)

### Always Do
- Always show data provenance: "Based on 50,000 simulations" / "Weather from OpenWeatherMap"
- Always update probabilities after every completed match
- Always attribute the methodology: transparent about what feeds the model
- Always test NRR calculations against known IPL data before shipping
- Always cache aggressively. Cricket APIs go down during live matches.
- Always design mobile-first. This is a second-screen product (phone while watching TV).
- Always include "last updated" timestamps on all data displays
- Always make share cards look premium. This is the viral loop. (Price at the Higher End.)
- Always push back if something looks wrong. Flag it, propose an alternative, then execute. (Creative Challenger.)
- Always validate data before trusting it. If an API returns garbage, don't render garbage. (Be Objective.)
- Always have a fallback. Primary API down? Fallback API. Fallback down? Cached data. Cached data stale? Show "last updated 2 hours ago" honestly. (RDJ Rule: there's always a second act.)

### Performance
- Target: <2s initial load, <500ms for simulation results (pre-computed, cached)
- Run full simulation after each match completion, cache results in Supabase
- On-demand scenario simulation should return in <3s (reduce to 10K sims for interactive)
- Lazy load match deep-dive data (don't fetch weather for all 70 matches upfront)

## Content & Voice

### The Win Path Voice
- Knowledgeable friend, not a textbook
- Specific, not vague ("win by 30+ runs vs GT" not "improve your NRR")
- Second person always ("your team needs" not "the team needs")
- Strategic, not just statistical ("this is your must-win because..." not just "62% win probability")
- Cricket-literate: use terms like powerplay, death overs, middle overs, dew factor, NRR naturally
- Never condescending to casual fans. Explain NRR when first shown, then trust they know.

### Social Content Templates (for daily posting)
- "RCB Win Path Update: 3 wins in 5. Your Top 4 odds just jumped to 67%. Here's what you need this week..."
- "Tonight's match swings your season. MI vs CSK isn't just their fight, it's YOUR fight. Here's why..."
- "NRR Alert: If PBKS chase this in 18 overs, they leapfrog your team. Watch the run rate."
